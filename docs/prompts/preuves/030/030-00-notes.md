# Preuves 030 — staging (:8001), Base de données → Composants

## Reproduction du 422 (detail = tableau Pydantic)
Le vrai backend renvoie un **422** avec `detail` sous forme de **tableau** d'objets `{type, loc, msg}` sur une requête invalide, ex. `GET /api/bom/components?limit=-5` :
```
{"detail":[{"type":"greater_than_equal","loc":["query","limit"],
            "msg":"Input should be greater than or equal to 1","input":"-5","ctx":{"ge":1}}]}
```
Pour déclencher ce cas **dans le panneau lui-même**, le paramètre `limit` de la requête `/bom/components` a été temporairement forcé à `-5` (patch de l'URL XHR côté navigateur) : la réponse **422 réelle** du serveur remonte alors dans le `catch` du composant.

## Captures
- `030-01-composants-charge-ok.jpg` — onglet **Composants** chargé normalement (1025 composants), aucun ErrorBoundary.
- `030-02-422-message-propre-sans-crash.jpg` — après le 422 tableau : le panneau affiche un **message texte propre** « **Input should be greater than or equal to 1** » (le `msg` extrait par `extractApiError`), **sans** erreur React #31, **sans** ErrorBoundary « Une erreur est survenue — Base de données », et le panneau reste **fonctionnel**.

Avant le correctif, ce même 422 (tableau d'objets) était injecté brut dans un `<Alert>` → **React #31** (« Objects are not valid as a React child ») → white-screen de toute la section via l'ErrorBoundary.
